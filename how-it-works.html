1

Dataset

src/data.py

A model learns by example, so first we need examples where the answer is already known. We download the IMDB Large Movie Review Dataset — 50,000 real movie reviews from Stanford, each already tagged positive or negative.

In
one line of code: load_dataset("stanfordnlp/imdb")
→
Out
25,000 train reviews (to study)
25,000 test reviews (the honest exam)
Why split it in half? Grading the model on reviews it already studied would be like giving a student an exam containing the exact questions they memorised. The test half is never studied — so the score is honest.
2

Data preprocessing

src/train.py — inside the TF-IDF step

Raw text is messy: capital letters, punctuation, filler words. Before the model can learn, the text is cleaned and chopped into pieces (called tokens).

In
"This Movie Was a TOTAL waste of time!!!"
→
Out
["this", "movie", "was", "total", "waste", "of", "time"]
lowercased · punctuation dropped · split into words

We also keep word pairs (bigrams) like "waste of" and "not good" — because "not good" means the opposite of "good", and single words alone would miss that.

Nice detail: IMDB is already clean and perfectly balanced (50% positive / 50% negative), so we don't need to fix missing values or rebalance classes. That's one of the reasons this dataset was chosen.
3

TF-IDF vectorization (feature engineering)

src/train.py — TfidfVectorizer

Computers can't read words — only numbers. TF-IDF turns each review into a long list of numbers, one slot per word. The clever part is how it scores each word:

PartWhat it meansEffect
TF — term frequencyhow often the word appears in this reviewrepeated words count more
IDF — inverse document frequencyhow rare the word is across all 25,000 reviewscommon words get crushed
In
["this","movie","was","total","waste","of","time"]
→
Out
the · movie · was → ≈ 0.00
waste → 0.61   total → 0.28
Plain words: TF-IDF automatically highlights the words that carry an opinion ("waste", "brilliant") and ignores the filler ("the", "movie", "was") — without anybody writing a list by hand. That's the feature engineering.
4

Logistic Regression (training)

src/train.py — python -m src.train

Now the actual learning. Logistic Regression is the simplest useful "brain": it learns one weight per word. It reads all 25,000 training reviews and nudges each weight up or down until its guesses match the known answers.

WordLearned weightMeaning
worst−3.1strong push toward "negative"
waste−2.7push toward "negative"
excellent+2.8push toward "positive"

To score a new review it just adds up the weights of the words present, then squashes the total into a probability between 0 and 1.

In
25,000 reviews + their known answers
→
Out
a trained model that outputs P(negative) — e.g. 0.92 = "92% sure this is negative"
Why so simple? Because it's fast (~2 ms), runs on any laptop with no GPU, and — most importantly — you can read the weights and see exactly why it decided. A giant AI model can't tell you that.
5

Evaluation

src/evaluate.py

Now we grade it — on the 25,000 test reviews it has never seen. We deliberately do not use plain "accuracy", because accuracy lies: if 90% of reviews were positive, a lazy model that always says "positive" scores 90% while catching zero angry customers.

MetricThe question it answersOur score
RecallOf all truly angry reviews, how many did we catch?0.896
PrecisionOf the ones we flagged, how many really were angry?0.904
F1A fair blend of both — one number0.900
Precision@1000Of the 1,000 we're most sure about, how many are right?1.00

The project's success gate was F1 ≥ 0.85. We scored 0.90 — passed with room to spare.

Precision@1000 = 1.00 is the business win: support works the top of the queue first. This says the top 1,000 flagged reviews are all genuinely negative — zero wasted agent time where it matters most.
6

MLflow — experiment tracking

src/train.py — mlflow.log_param / log_metric / log_artifact

This is what turns a school exercise into an industrial project. Every training run automatically writes its settings, its scores, and the model file into a logbook you can browse in a web dashboard.

MLflow recordsExample from our project
Parameters (the settings)max_features=50000, C=1.0, ngram_range=1-2
Metrics (the scores)f1_negative=0.900, recall=0.896
Artifacts (the outputs)the trained tfidf_logreg.pkl file
Run ID (the fingerprint)a unique id for every single run

Why it matters: six months from now, someone asks "which settings produced the 0.90 model, and can we rebuild it?" Without MLflow that's guesswork. With it, you open the dashboard, read the exact parameters, and reproduce it.

See it yourself: mlflow ui --backend-store-uri sqlite:///mlflow.db → open 127.0.0.1:5000, and compare every run side by side.
7

Saved model

models/tfidf_logreg.pkl

Training takes a couple of minutes — far too slow to redo for every review. So the finished model is frozen into a single file on disk. The live service loads that file once at startup and then answers instantly.

In
the trained model, in memory
→
Out
models/tfidf_logreg.pkl — loads in milliseconds, forever
Train once, predict millions of times. This split — slow training offline, fast prediction online — is the core shape of every production ML system.
8

FastAPI — the live service

service/app.py

A model sitting in a file helps nobody. FastAPI wraps it in a small web server so the website can ask it questions over the internet.

# the website sends:
POST /review   { "text": "Charged twice, cancelling!!!" }

# it answers in ~2 milliseconds:
{ "p_negative": 0.80, "urgency": 0.99, "route": "support_urgent" }
Try it live: run uvicorn service.app:app --port 8000, then open 127.0.0.1:8000/docs — FastAPI builds a free interactive test page. Paste any review, hit Execute, see the answer.
9

Prediction — the two scorers

src/predict.py · src/urgency.py

Every incoming review is scored twice, because "angry" and "urgent" are different questions:

Scorer 1 — Sentiment (the ML model): is this review negative? → P(negative) = 0.80

Scorer 2 — Urgency (transparent rules): is it an emergency? → looks for churn words (cancel, refund), money words (charged, billing), anger words, and shouting (!!!, ALL-CAPS) → urgency = 0.99

Why rules and not ML for urgency? A support manager must be able to see exactly why a review was escalated. Our scorer returns the literal words that fired. A black box saying "trust me, it's urgent" is useless in a real support workflow.
10

Business routing — the decision

src/urgency.py — route() · src/threshold.py

Finally, the two scores become an action. This is where machine learning turns into business value:

SentimentUrgencyWhere it goesWhy
NegativeHighsupport — nowa customer is walking out the door
NegativeLowfeedback backloguseful criticism, read it weekly
Positive—analyticstrack trends, no action needed

And the cut-off for "negative" isn't the lazy 0.5 — it's chosen by money. Contacting costs $1, saving a customer is worth $9, so break-even precision is only 11% and the profit-optimal cut-off lands at 0.44.

The full journey: a Stanford dataset → cleaned text → numbers → a trained brain → an honest grade → a tracked experiment → a frozen file → a live web service → two scores → a support agent picking up the phone before the customer leaves.
