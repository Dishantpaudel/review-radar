1

Dataset

src/data.py

A supervised model learns from examples where the answer is already known. The system uses the IMDB Large Movie Review Dataset (Maas et al., 2011) — 50,000 reviews from Stanford, each labeled positive or negative.

In
load_dataset("stanfordnlp/imdb")
→
Out
25,000 train — the model learns from these
25,000 test — held back for measurement
Why the split is enforced: the model never sees the test half during training. Measuring on data the model has already fit would report memorisation as if it were skill. Every metric in this project comes from held-out data.
Known limitation. IMDB is a stand-in for production data. It is movie criticism, balanced 50/50, and contains no billing language. Real platform reviews run roughly 80% positive and include payment complaints the model has never seen. This has measurable consequences (see stage 9) and drives the top roadmap item: retrain on first-party review data.
2

Preprocessing

src/text.py · src/train.py

Raw text carries capitalisation, punctuation and filler that mean nothing to a linear model. One shared normaliser (src/text.py) lowercases, strips <br> markup, folds apostrophes, and flattens elongation before anything is tokenised.

In
"This Movie Was a TOTAL waste of time!!! I don't like it"
→
Out
["this","movie","was","total","waste","of","time","i","dont","like","it"]
lowercased · markup stripped · don't → dont · tokenised

Word pairs (bigrams) are retained alongside single words, because "not good" inverts "good" and single tokens cannot express that. Folding don't → dont is what lets the "dont like" bigram exist as one feature instead of splitting on the apostrophe.

Why one shared normaliser matters: the same function feeds both the model and the urgency engine, so they can never disagree about what a word is. It is baked into the saved model (stage 7), which makes it structurally impossible to preprocess differently at inference than at training — the classic train/serve skew simply cannot occur here.
Steps deliberately absent: no imputation and no class rebalancing. IMDB has no missing values and is already balanced, so adding those stages would be ceremony rather than engineering.
3

TF-IDF vectorisation — feature engineering

src/train.py

Linear models consume numbers, not language. TF-IDF converts each review into a sparse vector, scoring each term by two factors:

ComponentMeasuresEffect
TF — term frequencyhow often the term appears in this reviewrepetition increases weight
IDF — inverse document frequencyhow rare the term is across the corpusubiquitous filler collapses toward zero
In
["this","movie","was","total","waste","of","time"]
→
Out
the · movie · was → ≈ 0.00
waste → 0.61  ·  total → 0.28
The point: no engineer wrote a list of significant words. TF-IDF derives which terms carry opinion from corpus statistics alone. That derivation is the feature engineering.

Two vectorisers, unioned — words and characters

Whole-word features have one fatal gap: a word never seen in training contributes nothing at all, because it has no slot in the vector. Real customers misspell exactly the words that carry the sentiment, so "its borst" (worst) scored a confident 0.16 — positive under words alone. So a second vectoriser runs over character n-grams (3–5 letters) and its output is concatenated onto the word vector: 50,000 word features + 50,000 character features = a 100,000-value vector.

In
"borst" (a misspelling of "worst")
→
char n-grams
"bo·bor·ors·rst·st " — shares 3 of 5 chunks with "worst", so the model still sees most of the evidence
Measured, not assumed. On clean IMDB this adds only ~0.003 F1 — because IMDB is edited prose with almost no misspellings. The benefit lives on the short, mistyped reviews real customers write, where "its borst" moves from 0.16 (positive) to 0.88 (negative). src/robustness.py and tests/test_typos.py exist precisely because the headline F1 cannot see this.
4

Training

src/train.py — python -m src.train

Logistic regression learns one weight per feature. It passes over the 25,000 training reviews, adjusting weights until its predictions align with the known labels. Scoring a new review sums the weights of the features present and maps the total onto a probability.

z = intercept + Σ (tfidf[feature] × weight[feature])
P(negative) = 1 − sigmoid(z)

Word weights extracted from the trained model:

Strongest negative
bad −4.11worst −4.05"the worst" −3.34awful −2.96waste −2.83boring −2.68

Strongest positive
great +3.11"the best" +2.55best +2.30fun +2.26excellent +2.21
Two bigrams rank among the strongest features — "the worst" and "the best". That is direct evidence the decision in stage 2 to keep word pairs was worth its cost, and the experiment sweep quantifies it at +0.011 F1. Weights are smaller than in a word-only model because character n-grams (stage 3) now carry part of the signal, so "bad" is split across the word feature and its letter-chunks rather than riding on one weight.
5

Evaluation

src/evaluate.py

The model is measured on the 25,000 held-out reviews. Metrics target the negative class specifically; overall accuracy is not used.

Why accuracy is rejected: production traffic is roughly 80% positive. A model that answers "positive" unconditionally scores 80% accuracy while detecting zero at-risk customers. Accuracy would rank that model highly. It is the wrong instrument.
MetricQuestionResult
RecallOf all genuinely negative reviews, how many were detected?0.901
PrecisionOf those flagged, how many were genuinely negative?0.905
F1Balance of the two — the project's acceptance gate0.903
Precision@1000Of the 1,000 highest-confidence flags, how many are correct?1.00

The acceptance gate was F1 ≥ 0.85. The model reaches 0.903.

Precision@1000 = 1.00 is the operational result. Support capacity is finite and the queue is worked top-down. Every one of the first thousand flagged reviews is genuinely negative, so no agent time is spent on false alarms where capacity is scarcest.

Model persistence

models/tfidf_logreg.pkl

Training takes roughly seven minutes — the character n-grams turn 25,000 reviews into millions of letter-chunks — far too slow to repeat per request. The fitted pipeline is serialised to a single file. The service loads it once at startup and reuses it for every subsequent prediction.

In
the fitted pipeline, in memory
→
Out
models/tfidf_logreg.pkl — loads in milliseconds
Train once, predict continuously. This asymmetry — expensive training offline, cheap inference online — is the defining shape of a production ML system. The vectoriser travels inside the same pickle as the classifier, which eliminates train/serve skew: it is structurally impossible to apply different preprocessing at inference than at training.

Prediction — two independent scorers

src/predict.py · src/urgency.py

Each review is scored twice, because "negative" and "urgent" are different questions and a single model answers only the first.

Scorer 1 — Sentiment (learned). Is the review negative? → P(negative)

Scorer 2 — Urgency (rule-based). Is the customer leaving? → scans for churn intent (cancel, refund, "dont want my subscription"), payment terms (charged, billing), anger vocabulary, and shouting (!!!, capitalisation). Returns a score plus the literal terms that matched.

Why urgency is rule-based rather than learned

Three reasons, in order of weight:

  1. No labels exist. IMDB carries sentiment labels, not urgency labels. Training a supervised urgency model would require inventing the ground truth.
  2. Escalation must be auditable. A support lead needs to see why a ticket jumped the queue, and a customer must be able to contest it. The engine returns the exact matched terms.
  3. It is in-domain where the model is not. The sentiment model has never seen billing language; the rules were written for precisely that vocabulary.
The domain gap, measured. The same system on two inputs:
InputP(negative)Interpretation
"waste of time, boring, awful acting, terrible ending"1.0000movie vocabulary — confident and correct
"Charged twice, I want a REFUND, cancelling today!!!"0.165billing vocabulary — out of domain, unreliable

The second review is unmistakably from a departing customer, yet the sentiment model rates it as probably positive — even more so now that character n-grams pulled its billing-language score down to 0.165. It has no billing vocabulary to reason with. The urgency engine scored the same text at 0.842. This is the concrete reason the two scorers exist: each covers the other's blind spot. (That 0.165 is also why the routing floor in stage 10 has to sit at 0.1, not 0.2 — a higher floor would let this unreliable score veto the reliable one.)

10

Business routing

src/urgency.py — route() · src/threshold.py

The two scores resolve into an action. This is where a prediction becomes a business decision.

if urgency >= 0.4 and p_negative >= 0.1:  return "support_urgent"
if p_negative >= 0.5:                    return "feedback_backlog"
return "analytics"
ConditionRouteRationale
Urgent, and not clearly positivesupport_urgentthe customer is leaving — respond within hours
Negative, calmfeedback_backloglegitimate criticism — review on a weekly cycle
Neitheranalyticstrend reporting, no intervention

Why urgency escalates independently

Explicit churn intent is a support event regardless of the sentiment score. Sentiment acts only as a low floor (≥ 0.1), and even that job is mostly held upstream: negation scope in the urgency engine strips a churn word inside praise — "I'd never cancel, I love it" — before it ever reaches the router. The floor's sole remaining duty is a churn word inside confident praise (p_negative < 0.1), the one regime where this movie-trained model is reliable.

This ordering was originally wrong, and the defect was real. The first implementation evaluated sentiment first and returned analytics whenever P(negative) < 0.5. Because the sentiment model is out of domain on billing language, it scored a customer writing "Charged twice, I want a REFUND, cancelling today!!!" at 0.165 — and that unreliable score silently overruled an urgency of 0.842, routing a departing customer to analytics. An unreliable component must not be able to veto a reliable one. The fix escalates on urgency independently; because the guarding floor sits only 0.1 above where the model scores such reviews, regression tests now drive the real model on four phrasings rather than trusting a constant.

Where the 0.5 threshold comes from

It does not come from the model — it comes from the economics. Outreach costs $1; a retained customer is worth $9. Break-even precision is therefore $1/$9 ≈ 11%, and sweeping every threshold against expected profit puts the optimum at 0.43, below the conventional 0.5. The threshold is a business parameter, recomputed whenever costs change.

End to end: a public dataset → normalised text → a sparse vector → a fitted linear model → a held-out measurement → a tracked experiment → a serialised artifact → an HTTP service → two independent scores → a support agent contacting a customer before they leave.
