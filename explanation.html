review-radar, explained simply

You said you don't have the idea yet — so this walks through every file and every decision in plain words, as if explaining to a friend. No jargon left unexplained.

github.com/Dishantpaudel/review-radar

What is this project?

A movie website gets hundreds of thousands of reviews a month. Buried in that pile, real customers are saying "I was charged twice, I'm leaving." Nobody can read them all, so those people slip away unnoticed.

This project builds a robot reader. Every review that arrives passes through it in about 2 milliseconds. The robot decides two things: (1) is this review angry? and (2) is it an emergency or just calm criticism? Then it sends the emergencies to a human support agent immediately, and files the rest where they belong.

In one line: read every review the second it arrives, catch the angry-and-about-to-leave customers, and get a human to them before they go.

The big picture

Everything in the project serves this one flow. Read it left to right:

a new review
   │
   ▼
[ 1. Sentiment model ]   →  P(negative) = a score from 0 to 1
   │                         (0.92 means "92% sure this is negative")
   ▼
[ 2. Urgency scorer  ]   →  urgency = 0 to 1
   │                         (is it an emergency, or calm feedback?)
   ▼
[ 3. Router ]
   ├─ negative + urgent  →  support team, now
   ├─ negative + calm    →  product feedback backlog
   └─ positive           →  analytics dashboards

Every file you're about to read is one piece of this pipeline — either building a piece, testing a piece, or showing a piece working.

How to read the flow: data.py gets the reviews → train.py learns from them → evaluate.py checks the grade → threshold.py & urgency.py add the two smart decisions → predict.py & app.py put it all together for real use. Tests and notebooks prove each step works.

1 · data.py get the reviews

What it does: downloads and loads the movie reviews.

A model learns by example, so first we need labeled examples. This file fetches the famous IMDB dataset: 50,000 real movie reviews, each already tagged "positive" or "negative."

Plain words: it's like handing a new employee 50,000 reviews that are already marked "happy" or "angry," so they can learn the pattern.

Key idea: train vs test

The 50,000 reviews are split in half:

  • 25,000 train — the model studies these.
  • 25,000 test — the model is graded on these, which it has never seen.

Why split? Testing on reviews the model already studied is like giving a student an exam with the exact questions they memorized — the score would be a lie. The test set is the honest exam.

Label meaning throughout the project: 0 = negative, 1 = positive.

2 · train.py teach the model

What it does: turns reviews into a working sentiment model, and saves it.

This is where the learning happens. It has two parts that work in a chain (a "pipeline"):

Part A — TF-IDF: turn words into numbers

Computers can't read words, only numbers. TF-IDF gives each word a number: high if the word is frequent in this review but rare across all reviews.

  • "the", "movie", "and" → near zero (they're everywhere, they tell you nothing)
  • "refund", "masterpiece", "boring" → high (rare and meaningful)
Plain words: TF-IDF highlights the words that actually carry an opinion and ignores the filler.

Part B — Logistic Regression: the simple brain

It learns a weight for every word. Angry words get negative weights, happy words get positive weights. To score a review, it adds up the weights of the words present, then squashes the total into a probability between 0 and 1 — that's P(negative).

WordLearned weightMeaning
worst−3.1strong "negative" pull
boring−2.4negative pull
excellent+2.8positive pull
Bonus: because it's this simple, you can literally read why it decided — the word weights are right there. A big AI model can't easily tell you that.

It also records everything in MLflow

Every time it trains, it logs the settings and scores to a tracking tool called MLflow, so you can compare experiments later in a web dashboard. Finally it saves the trained model to models/tfidf_logreg.pkl so the live service can load it instantly.

3 · evaluate.py grade the model

What it does: measures how good the model is, using the right yardsticks.

We don't measure plain "accuracy," because accuracy hides failures. If 90% of reviews were positive, a lazy model that says "positive" every time would be 90% accurate while catching zero angry customers. So we measure the negative class specifically:

MetricPlain meaningOur score
RecallOf all the truly angry reviews, how many did we catch?0.896
PrecisionOf the ones we flagged as angry, how many really were?0.904
F1A fair blend of the two (one number to rule them)0.900
Precision@1000Of the 1,000 we're most sure about, how many are right?1.00

The project's success gate was F1 ≥ 0.85. We hit 0.90 — passed. And precision@1000 = 1.00 means the very top of the support queue is pure signal: no wasted effort where it matters most.

Why precision@k matters: support has limited time. They work the top of the queue first. This metric says the top of the queue is trustworthy.

4 · compare_transformer.py test a fancier model

What it does: checks whether a big modern AI model beats our simple one.

We tested DistilBERT — a small, fast relative of the AI models behind ChatGPT, already trained to judge sentiment. We used it straight off the shelf (no extra training), scored 3,000 test reviews, and measured the same metrics.

The result

DistilBERT: F1 = 0.898
Our simple model: F1 = 0.900
A tie.

The decision

Ship the simple model. Same quality, ~100× cheaper to run, and explainable. The fancy one is a future upgrade, not the starting point.

Why the tie is a great result: a senior engineer doesn't reach for the biggest tool by default. Proving the cheap tool is good enough is the smart, mature call — and now you can defend it with numbers.

5 · threshold.py the money math — feedback #2

What it does: picks the cut-off score for "act on this" based on profit, not model quality.

Full explanation in the Feedback #2 section below. In short: it tries every possible cut-off from 0.01 to 0.99 and computes the expected monthly profit at each, then picks the most profitable one. This is the direct answer to your teacher's second point.

6 · urgency.py emergency detector — feedback #1

What it does: decides if an angry review is an emergency or just calm criticism.

Full explanation in the Feedback #1 section below. In short: after the sentiment model says "this is negative," this file checks for emergency signals (cancel, refund, billing words, shouting) and routes only the true emergencies to support. This is the direct answer to your teacher's first point.

7 · predict.py score one review

What it does: the glue — takes one review, runs both scorers, returns the decision.

This file loads the saved model once, then for any review text it returns a small package: the negativity score, the urgency score, which signals fired, and the final route. It's what the live service calls for every incoming review.

# what score_review("Charged twice, cancelling!!!") returns
{
  "p_negative": 0.80,
  "urgency": 0.99,
  "urgency_signals": { "churn_phrases": ["cancelling","refund"], ... },
  "route": "support_urgent"
}

8 · service/app.py the live web service

What it does: exposes the model as a real API the website can call.

A small web server built with FastAPI. The website sends a review to it over the internet, and it replies in ~2 milliseconds. This is what makes the project real-time instead of a notebook experiment.

It offers two endpoints:

  • GET /health — "are you alive?" → {"status":"ok"}
  • POST /review — send {"text": "..."}, get back the full decision.
The demo trick: FastAPI gives you a free interactive test page at /docs. Open it in a browser, paste a review, and watch the answer — no coding needed. Great for showing the teacher live. (See How to run.)

9 · tests/ automatic checks

What it does: 10 small programs that prove the code behaves correctly.

Tests are safety nets. Run pytest and it checks, automatically:

  • test_urgency.py — an angry churn review scores urgent; calm feedback doesn't; routing sends each to the right place.
  • test_threshold.py — the money math is correct, and (the key claim) the profit-optimal cut-off really does land below 0.5.
  • test_api.py — the live API returns the right route for angry, calm, and positive reviews.

All 10 pass. If you ever change the code and break something, the tests catch it instantly.

Teacher Feedback #1 — "Negative is not the same as urgent"

Your teacher's exact concern: many negative reviews are constructive feedback, not emergencies. Some frustrated customers need a reply now. The system must tell them apart — otherwise you drown the support team.

How the project answers it

After the sentiment model flags a review as negative, urgency.py runs a second, transparent check. It's deliberately rule-based (not AI) for one reason: a support manager must be able to see exactly why a review was escalated. A black box that just says "trust me, it's urgent" is useless in a real support workflow.

The four signal groups it looks for

Signal groupExample wordsWeight
Churn intent (leaving)cancel, refund, never again, switching toheaviest
Money problemscharged, billing, took my moneyhigh
Angerscam, unacceptable, worstmedium
Shoutinglots of !!! and ALL-CAPS wordslow

It combines these into an urgency score from 0 to 1, and shows which exact words fired. Then the router decides:

SentimentUrgencyWhere it goesWhy
NegativeHighsupport — nowa customer is leaving; act fast
NegativeLowfeedback backloguseful criticism, read it weekly
Positive—analyticstrack trends, no action needed

Seen live

ReviewUrgencyRoute
"Charged twice, I want a REFUND, cancelling today!!!"0.99support — now
"The pacing dragged and the ending felt rushed."0.00feedback backlog
"Wonderful film, loved every minute."0.02analytics
The answer in one sentence: "who actually needs intervention?" → the negative reviews that also show churn intent, money problems, or raw anger — and the system now separates exactly those.

Teacher Feedback #2 — "Optimize the threshold on money"

Your teacher's exact point: contacting a customer costs only $1, so instead of only contacting the ones you're very sure about, maybe you should contact more people — and pick the cut-off that makes the most money, not the best F1.

First, what is a "threshold"?

The model outputs P(negative) from 0 to 1. The threshold is the line you draw: "if the score is above X, we act." The lazy default is 0.5. But 0.5 isn't sacred — it's just a starting guess.

The money logic

Cost of contacting one customer: C = $1

Value of saving one customer: win-back rate × monthly revenue × months kept = 0.15 × $10 × 6 = $9

Because the prize ($9) is 9× the cost ($1), you make money even if most of your contacts are wrong. The exact break-even point:

break-even precision = C / (r·ARPU·M) = $1 / $9 ≈ 0.11

So you profit as long as at least 11% of the people you contact were genuinely about to leave. That's a very low bar — which means you should cast a wide net.

What threshold.py actually does

It sweeps every threshold from 0.01 to 0.99, and at each one computes:

profit = (customers saved × $60) − (people contacted × $1)

Then it picks the threshold with the highest profit. Results on our model:

ThingValueMeaning
Break-even precision0.11the profit floor
Default threshold0.50the lazy guess
Profit-optimal threshold0.44below 0.5, as predicted
Precision at that point0.878× above break-even → very safe
The punchline your teacher wanted: because contacting is cheap and saving is valuable, the best threshold is lower than 0.5 — you should flag more reviews, not fewer. Model-quality metrics (F1) and business-value metrics choose different thresholds, and the business one wins.
Honest footnote (say this — teachers love honesty): on our particular model the extra profit from tuning is small (~$1.9k/month), because the model is very confident and rarely outputs middle-range scores. But the method is the real deliverable: when costs change, or the model is noisier, or real data is imbalanced (real reviews are ~80% positive vs 50% here), you just re-run the sweep and get the new best threshold. You've built the tool, not just one number.

How to run & demo it

Everything runs from inside the review_model folder with your virtual environment active.

Option A — just look (no running)

1
Open the notebooks. In VSCode, open notebooks/02_baseline_model.ipynb and 04_business_threshold.ipynb. They're already run — scroll to see the scores and charts.
2
Read the README on GitHub — it's the whole story on one page.

Option B — see the experiments dashboard

cd review_model
mlflow ui --backend-store-uri sqlite:///mlflow.db --port 5001

Open http://127.0.0.1:5001 → two experiments with all the logged scores.

Option C — the live demo (best for showing the teacher)

1
Start the server:
cd review_model
uvicorn service.app:app --port 8000
2
Open the free test page: in a browser go to http://127.0.0.1:8000/docs
3
Click POST /review → Try it out → paste a review → Execute. Try "Charged twice, I want a refund, cancelling today!!!" and watch it return support_urgent with the signals that fired. Then try "the pacing was a bit slow" and watch it go to feedback_backlog. That's feedback #1 proven live.

Option D — rebuild from scratch (prove it's reproducible)

cd review_model
python -m src.train              # retrains, ~2 minutes
python -m src.compare_transformer # optional, ~10 min on CPU
pytest                            # all 10 tests pass
