SENTIMENT · NLP · MLOps1 / 16
The Problem

Angry customers are hidden in the reviews

  • A movie website gets ~300,000 reviews a month.
  • Nobody can read them all, so angry reviews are seen too late — after the customer already left.
  • An angry review is an early warning. Today it is ignored.
  • Goal: catch the warning in seconds, not weeks.
~95%
of consumers read reviews before choosing a product

Every ignored complaint pushes a customer closer to leaving — and pushes future customers away too.

The Data

IMDB Large Movie Review Dataset

  • From Stanford (Maas et al., 2011) — real IMDB reviews.
  • 50,000 reviews, labeled positive / negative.
  • Split 25k train / 25k test, perfectly balanced (50/50).
  • Loaded in one line, downloads on first run.
from datasets import load_dataset load_dataset("stanfordnlp/imdb")

Why it fits

  • Big, clean, balanced — almost no cleaning.
  • The most-trusted benchmark for text models.
  • Movie reviews for a movie site.

Honest note: a stand-in. Real reviews are ~80% positive and include billing complaints. Production plan: retrain monthly on the site's own reviews.

EVALUATION7 / 16
Model 2 · The Modern Alternative

Does a big AI model do better?

  • Tested DistilBERT — a small, fast cousin of the models behind ChatGPT, already trained for sentiment (SST-2).
  • No training needed — used it straight off the shelf, so it runs on CPU.
  • Scored 3,000 random test reviews and measured the same metrics.

Result: a tie

F1 = 0.898 vs the baseline's 0.900. Fine-tuning it on IMDB could reach ~0.93, but that needs a GPU.

What is DistilBERT?

It reads a sentence in context — it knows "not good" is negative even though "good" is a positive word. The baseline sees words mostly one at a time.

More powerful, but ~100× more expensive to run and a black box — you can't easily see why it decided.

DECISION9 / 16
Teacher Feedback #1 · Answered

Negative is not the same as urgent

A second scorer runs after sentiment. Rule-based on purpose, so support can see exactly why a review was escalated.

Signals it looks for

  • Churn (heaviest): cancel · refund · never again · switching to
  • Money: charged · billing · took my money
  • Anger: scam · unacceptable · worst
  • Shouting: lots of !!! and ALL-CAPS

The routing matrix

SentimentUrgencyRoute
NegativeHighsupport — now
NegativeLowfeedback backlog
Positive—analytics

Live example: "Charged twice, I want a REFUND, cancelling today!!!" → urgency 0.99 → support — now

FEEDBACK #110 / 16
Teacher Feedback #2 · Answered

Choose the threshold for money, not F1

The cut-off score for "act on this" was 0.5 by default. But the economics say otherwise:

break-even precision = C / (r·ARPU·M)
= $1 / $9 ≈ 0.11
  • You profit even if 89% of contacts are "wasted."
  • So the model should flag more reviews, not fewer.

What the sweep found

Break-even precision0.11
Default threshold0.50
Profit-optimal threshold0.44
Precision there0.87
Safety margin vs break-even8×

Optimal threshold landed below 0.5 — exactly the direction predicted. Threshold is now a business decision, re-run whenever costs change.

FEEDBACK #211 / 16
Feedback #2 · The Picture

Profit vs threshold

  • Contact more people → catch more at-risk customers → more profit, until precision drops toward the 11% break-even.
  • The peak sits left of 0.5.
  • On this very confident model the gain is modest (~$1.9k/mo); with noisy or imbalanced real data the gap grows. The method is the deliverable.

The Live Service

One API call, one decision

A small web server (FastAPI). Send a review, get back the score, the urgency, and the route.

# request POST /review { "text": "Charged twice, cancelling!!!" } # response (~2 ms) { "p_negative": 0.80, "urgency": 0.99, "route": "support_urgent" }
