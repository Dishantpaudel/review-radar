The Problem

Churn signals arrive as text — and nobody reads them

  • The platform receives ~300,000 reviews / month.
  • Human review does not scale. Complaints surface weeks late — after the customer already left.
  • Each angry review is a churn signal with a timer on it.
  • The window to save a customer is hours, not weeks.

The cost of doing nothing: every unanswered complaint is a customer walking out, plus a public review that deters the next buyer.

PROBLEM2 / 23
Why Machine Learning at all?

Because the alternatives break at scale

❌ Humans read them

300k reviews × 30s each = 2,500 hours/month. That's ~15 full-time staff, and they'd still be days behind.

❌ Keyword blacklist

"bad" fires on "not bad at all!". Misses sarcasm, misses phrasing nobody predicted. Endless manual upkeep.

✅ Learn from data

The model reads 25,000 labeled examples and derives the patterns itself — including ones no human would list.

The honest nuance: we still use rules — but only where rules are better. Sentiment is learned (ML); urgency is rule-based (transparent, auditable). Right tool, right job.

Step 1 · The Data

IMDB Large Movie Review Dataset

  • Stanford, Maas et al. 2011 — the field's standard sentiment benchmark.
  • 50,000 real reviews, labeled positive / negative.
  • 25k train / 25k test, perfectly balanced 50/50.
  • Only strong opinions kept (≥7★ or ≤4★) → clean labels.
from datasets import load_dataset load_dataset("stanfordnlp/imdb") # src/data.py

Why the train/test split is sacred

The model never sees the 25k test reviews during training. Grading on studied data would be an exam with the answers leaked. Every number in this deck comes from unseen data.

Honest limitation

IMDB is a stand-in: movie reviews, balanced 50/50. Real platform reviews are ~80% positive and contain billing complaints the model never saw. Production plan: retrain monthly on our own reviews.

Step 4 · Inside the Model

Weights it actually learned

Logistic Regression learns one weight per feature. Scoring = add the weights of the words present, squash into 0→1.

# the whole model, conceptually z = intercept + Σ (tfidf[word] × weight[word]) P(negative) = 1 − sigmoid(z)

Why NOT accuracy

In production ~80% of reviews are positive. A model that answers "positive" every single time scores 80% accuracy — and catches zero angry customers. Accuracy rewards the useless model. So we measure the negative class only.

Why precision@k is the real KPI

Support has finite hours; they work the queue top-down. Precision@1000 = 1.00 means the first thousand tickets are all genuinely angry. Zero wasted agent time where it counts.

Benchmark · The Modern Alternative

We tested the transformer properly

  • DistilBERT (SST-2) — a distilled BERT, same family as the models behind modern chatbots.
  • Used pretrained, no fine-tuning → runs on CPU, no GPU bill.
  • Scored 3,000 random test reviews, identical metrics, logged to MLflow.

Why bother, if we ship the simple one?

Because "we didn't try it" is not an answer. Now we can say: we measured it, and the gain didn't justify the cost. That's the difference between a preference and a decision.

What DistilBERT does differently

It reads words in context — it knows "not good" is negative, and handles phrasing it never literally saw. Our baseline approximates this with bigrams.

The catch: 66 million parameters vs our ~50k weights. ~100× slower per review, and it cannot tell you why it decided.

DECISION14 / 23
Review Feedback #1 · Answered in code

"Negative" and "urgent" are different questions

Most negative reviews are constructive criticism — valuable, but not an emergency. Routing all of them to support drowns the team and buries the real fires.

A second scorer, deliberately rule-based

SignalExamplesWeight
Churn intentcancel · refund · never again0.45
Moneycharged · billing · overcharged0.30
Angerscam · unacceptable · fraud0.15
Shouting!!! · ALL-CAPS ratio0.10

Routing matrix

SentimentUrgencyRoute
NegativeHighsupport — now
NegativeLowfeedback backlog
Positive—analytics

Why rules, not ML, here?

A support lead must see exactly why a ticket jumped the queue. Our scorer returns the literal matched words. A black box saying "trust me, it's urgent" is unusable in an operational workflow — and impossible to appeal.

FEEDBACK #1 · URGENCY ENGINE15 / 23
Review Feedback #2 · Answered in code

The threshold is a business decision, not an ML one

The cut-off for "act on this" defaulted to 0.5 — an arbitrary convention. The economics say otherwise:

break-even precision = C / (r · ARPU · M)
= $1 / (0.15 × $10 × 6) = $1/$9 ≈ 0.11
  • Contact costs $1. A save is worth $9.
  • We profit even if 89% of outreach is "wasted."
  • ⇒ the model should flag more, not fewer.

What the sweep found src/threshold.py

Optimum landed below 0.5 — exactly the predicted direction. Re-run the sweep whenever costs change and the threshold updates itself.

Step 8–9 · Deployment

The model behind an API

A model in a notebook helps nobody. FastAPI turns it into a service the platform can call on every review event.

# the platform sends POST /review { "text": "Charged twice, cancelling!!!" } # answer in ~2 ms { "p_negative": 0.8041, "urgency": 0.989, "urgency_signals": { "churn_phrases": ["cancel","refund"] }, "route": "support_urgent", "latency_ms": 2.32 }

Production details that matter

  • Model loads once at startup (lifespan hook), not per request — that's the 2 ms.
  • Typed contract via Pydantic — malformed input rejected at the door.
  • /health endpoint — load balancers need a heartbeat.
  • Latency returned in every response — you cannot monitor what you don't measure.

The seven questions

What problem?

Detect negative and urgent reviews in real time so support acts before the customer churns.

Why ML?

300k reviews/month can't be read by hand; keyword rules break on negation and sarcasm.

Why this model?

TF-IDF + LogReg balances accuracy (F1 0.90), latency (2 ms), cost (CPU) and explainability.

Why compare DistilBERT?

To test whether a transformer earns its cost. It didn't — tie at ~100× the price.

What does MLflow do?

How does it work end to end?

Data → preprocess → train → evaluate → MLflow → persist → FastAPI → two scorers → business routing.

What makes it production-oriented?

Automated tests, experiment tracking, a deployed API, business-driven thresholds, a reproducible pipeline, and a documented retraining plan.
